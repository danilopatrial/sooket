## What this tests
Verifies that when the user interacts with one node's configuration and then clicks a different node of the same type, the second node's fields show its own saved data — not leftover values from the first node.

## Prerequisites
- App is running at http://localhost:3000
- A canvas with at least two nodes of the same type (e.g. two Text nodes, two If nodes) is open

## Steps — Text node data isolation

1. Add two **Text** nodes to the canvas
2. Set Node A's content to `"value A"`
3. Set Node B's content to `"value B"`
4. Click Node A — verify its Content field shows `"value A"`
5. Click Node B — verify its Content field immediately shows `"value B"` (not `"value A"`)
6. Click back to Node A — content still shows `"value A"`

## Steps — If node data isolation

7. Add two **If** nodes:
   - Node C: operator `== equals`, Compare To `"hello"`
   - Node D: operator `> greater than`, Compare To `100`
8. Click Node C — verify operator and Compare To field show `== equals` and `"hello"`
9. Click Node D — verify operator shows `> greater than`, Compare To shows `100`
10. No values from Node C bleed into Node D's display

## Steps — Anthropic node model selection

11. Add two **Anthropic** nodes:
    - Node E: model Haiku 4.5
    - Node F: model Sonnet 4.6
12. Click Node E — verify Haiku 4.5 is highlighted
13. Click Node F — verify Sonnet 4.6 is highlighted (not Haiku)
14. Verify the temperature slider is visible for Node E (Haiku) and hidden for Node F (Sonnet)

## Steps — unsaved state is scoped per node

15. Start typing in Node A's Text field (do not complete the change)
16. Click Node B — Node B shows its own content; the partial edit on Node A is not carried over
17. Click back to Node A — the partially typed change may or may not be preserved (depends on autosave timing), but it should not appear on Node B

## Expected result
- Node configuration is stored per-node in the React Flow `nodes` array
- Clicking a different node renders that node's own `data` prop — no cross-contamination
- The `onChange` callback is scoped to `updateNodeData(nodeId, patch)` for each specific node

## Failure indicators
- Clicking Node B shows the values that were configured for Node A
- The operator selected on one If node appears pre-selected on a different If node
- Temperature slider visibility on one Anthropic node is affected by another Anthropic node's model selection

## Severity rationale
High: stale state cross-contamination would cause silent misconfiguration — users might believe they've set a value for one node when they've actually set it for another.

## Source reference
`components/canvas/WorkflowCanvas.tsx` lines 618-639 (per-node `onChange` injection: `onChange: (p) => updateNodeData(n.id, p)` — each node's onChange is bound to its own `nodeId`), lines 583-595 (`updateNodeData`: patches only the specific node's data via `n.id === nodeId`).

## Notes
Since node configuration UI lives inside each node component (not in a separate selection panel), and data flows from the `nodes` array via props, switching nodes inherently shows different components with different data. The main risk would be if a node component cached data in local `useState` — the node components in this project derive all values from `data` props rather than local state.
