## What this tests
Verifies that binary data stored during workflow execution (e.g. from an HTTP Request node receiving a binary response) is retrievable at `GET /api/binary/[id]` with correct `Content-Type` and `Content-Length` headers, and returns 404 for unknown IDs.

## Prerequisites
- A workflow with an HTTP Request node configured to fetch a binary URL (e.g. `https://httpbin.org/image/png`) has run successfully and produced a binary reference ID in its output

## Steps — retrieving a valid binary reference

1. Run a workflow that returns a binary response (image, PDF, etc.) via `POST /api/v1/chat`
2. Capture the binary reference ID from the workflow output (an `IBinaryRef` object with `{ id, mimeType, fileName, size, backend }`)
3. Send a GET request:
   ```bash
   curl -si http://localhost:3000/api/binary/<id>
   ```
4. Verify HTTP status **200**
5. Verify `Content-Type` header matches the MIME type from the HTTP Request response (e.g. `image/png`)
6. Verify `Content-Length` header equals the byte size of the binary data (matches `ref.size`)
7. Verify the response body is the raw binary data (not base64, not JSON-wrapped)

## Steps — unknown ID returns 404

8. Send `GET /api/binary/nonexistent-id`
9. Verify HTTP status **404** with empty body

## Steps — expired memory entry returns 404

10. With the default in-memory backend, binary entries expire after **1 hour** (3,600,000 ms)
11. After 1+ hour, request the same ID — verify 404 (entry expired and cleaned up)

## Steps — ID format

12. Verify the binary reference ID is a UUID (format: `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`)
13. This ID is generated by `crypto.randomUUID()` in `BinaryDataService.write()`

## Expected result
- Valid in-memory ID: 200 response with raw binary body, `Content-Type: <mimeType>`, `Content-Length: <size>`
- Unknown or expired ID: 404 with empty body (`null`)
- Backend: default is "memory" (1-hour TTL); filesystem backend (`BINARY_DATA_BACKEND=fs`) has no TTL
- No CORS headers added (endpoint is not under `/api/v1/chat`)

## Failure indicators
- Valid ID returns 404 immediately after writing
- `Content-Type` does not match the MIME type detected by the HTTP Request node
- `Content-Length` does not match the actual byte length of the binary
- Response body is base64-encoded instead of raw binary

## Severity rationale
Binary files (images, PDFs) retrieved via HTTP Request nodes are unusable if the binary endpoint returns wrong headers or corrupts the data.

## Source reference
`app/api/binary/[id]/route.ts` lines 8-17 (`getRef` for metadata, `read` for buffer, response with `Content-Type: ref.mimeType` and `Content-Length: ref.size`); `lib/binary-data.ts` lines 28-54 (`write`: stores buffer with UUID, sets `expiresAt = now + 3_600_000` for memory), lines 56-68 (`read`: returns null/throws on expired), line 91 (singleton `binaryDataService`).

## Notes
The binary endpoint has no authentication — any client that knows a valid ID can retrieve the binary. IDs are UUIDs (128-bit random) so guessing is impractical. The endpoint is served at `/api/binary/[id]`, not under `/api/v1/`.
